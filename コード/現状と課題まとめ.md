# 人工知能判別ゲーム — 現状・設計判断・課題まとめ

作成日: 2026-07-04
対象: 開発の現状引き継ぎ、および設計判断の記録

---

## 0. このプロジェクトは何か

プレイヤーが「会話相手が人間かAIかを判別する」Webベースの社会的推理ゲーム。
プレイアブルなゲームであると同時に、**研究データ収集ツール**でもある。二重目的である点が
設計判断の根っこにある。

答えたい研究の問いは3つ。

1. AIのどんな発言が見破られやすいか
2. 人間は何を根拠に判断するか
3. どんな質問がAIをあぶり出すか

---

## 1. 確定している設計方針

- **第一実装軸**: 役割質問（インタビュー）式 × 非対称型1対1。
  質問者が質問を投げ、回答者が答え、一定ターン後に質問者が「相手は人間かAIか」を判定する。
- **包含関係の原則**: 大人数型が1対1を内包し、非対称型が対等型を内包する。
  単純な方から土台を作り、拡張で複雑な方へ広げる（対等型は非対称型を双方向化して作る）。
- **入り口は3つ**: シングル対戦 / クイックマッチ（公開部屋） / プライベートマッチ。
- **AI出現ルール（入り口ごと）**:
  - シングル: 相手がAIか人間かランダム
  - クイックマッチ: AI混在／全員人間をランダム。空き枠はAIで補填
  - プライベート: ホスト設定（ランダム／必ずAI／AIなし）
- **AI接続**: Gemini無料枠を第一候補（精度より安さ優先。安いモデルはむしろ判別難易度が上がる利点）。
  ローカルLLM／有料APIとの選択は保留。
- **開発方針**: 最小動作するものを先に作り、後から拡張するインクリメンタル型
  （Among Us / Jackbox 等の「最小構成で出し、人気後にモード追加」に倣う）。

## 2. 実装の現状

### 2.1 ファイル構成

| ファイル | 役割 |
|---|---|
| `app.py` | Flaskサーバ。HTTPルート（シングル）＋ Socket.IOイベント（オンライン） |
| `ai_backend.py` | AI応答の差し替え点。google-genai SDK、モデル `gemini-3.5-flash`。APIキー無しはダミー応答 |
| `database.py` | SQLite。5テーブルを `game_id` で連結。`export_json` / `export_csv` あり |
| `templates/index.html` | 画面遷移・チャットUI・判定・任意アンケート・オンライン処理 |
| `test_online.py` | オンライン経路の自動テスト（実DB／AIに触れない） |

### 2.2 データベース設計（SQLite・5テーブル）

- `games` … 1ゲームの構成・結果
- `players` … 各参加者（役割・投票先・正誤）
- `messages` … 会話ログ。`reply_to` で質問と回答を紐付ける
- `annotations` … `message_id` で個々の発言に注釈（決め手／違和感など）を紐付ける
- `surveys` … 判定後の任意アンケート（自信度・判断根拠・自由記述）

`reply_to`（質問-回答の紐付け）と `annotations`（発言レベルの注釈）を持たせたことで、
3つの研究の問いを同時に追える構造になっている。

### 2.3 シングル（従来・HTTP）

- ルート: `/start` `/ask` `/judge` `/survey` `/messages` `/annotate`
- 状態は Flask セッション（クッキー）に保持
- 相手は現状つねにAI（`/start` を `random.choice(["ai","human"])` にすれば相手人間も入る土台あり）
- 質問回数 `MAX_TURNS = 5`

### 2.4 画面遷移（`index.html`）

`showScreen()` による1画面ずつの切り替え方式。

```
タイトル(開始/終了)
 ├ 終了 → 終了画面
 └ 開始 → モード選択(シングル/グループ)
            ├ シングル → [Start / 戻る] → 準備スピナー → ゲーム
            └ グループ → ルーム作成(公開/プライベート)
                 ├ 公開      → クイックマッチ(マッチング) → ゲーム   ← オンライン実装済み
                 └ プライベート → 合言葉+人数上限 → ゲーム設定(AI数/AI側の人数) → ゲーム開始
                                                                        ↑ 準備中(未実装)
```

- 集めた設定は `config` オブジェクト（`entry` / `visibility` / `roomKey` / `maxPlayers` /
  `aiCount` / `humanAsAiCount`）に保持。将来オンライン化時にそのまま部屋設定として使える形。
- IME変換確定Enterでの誤送信対策済み（`compositionstart` / `compositionend`）。
- 「終了」はブラウザ制約（スクリプトが開いていないタブは `window.close()` で閉じられない。
  出典: MDN "Window: close() method" https://developer.mozilla.org/en-US/docs/Web/API/Window/close ）
  のため、終了メッセージ表示で代替。

### 2.5 オンライン（クイックマッチ・人対人2人）※今回の到達点

- 実装: Flask-SocketIO（`async_mode="threading"`）。追加インストールは `pip install flask-socketio`。
  起動は `socketio.run(app)`（従来の `app.run` から変更）。
- **状態はサーバー側メモリに部屋単位で保持**（`_waiting` / `_rooms` / `_sid_room` ＋ `threading.Lock`）。
  2人で1ゲームを共有するため、クライアント毎のセッションは使えない。
- サーバーが受け付けるイベント: `connect` / `join_quick` / `submit_question` /
  `submit_answer` / `submit_judge` / `disconnect`。
- 役割（質問者／回答者）はマッチ時にランダム割当。相手の正体は現状つねに `human`（AI補填は未実装）。
- DB保存は `entry_type="quick"`・人間2人で、シングルの `judge()` と同じ構造。
- 切断時は相手に `opponent_left` を通知。
- クライアント側: `index.html` に Socket.IO（cdn.socket.io 4.7.5）を読み込み、
  `onlineMode` フラグで `ask()` / `judge()` を Socket 経由に分岐。既存チャットUIを役割ごとに出し分け
  （質問者は入力可、回答者は質問到着まで入力ロック）。
- 検証: `test_online.py` で2クライアントの通しテスト（マッチング〜Q&A〜判定〜保存〜切断通知）が
  7項目全通過。

#### テストの根拠と限界（重要）

- 各チェックは `assert` で書かれており、条件が偽ならそこで停止する。7項目すべて表示された
  ＝すべての条件が成立した、という機械的事実。
- ただし `socketio.test_client` は**本物のクライアントではなく**、サーバーのコンテキスト内で動作し、
  発行イベントを保存するだけ。
  （出典: Flask-SocketIO "API Reference" https://flask-socketio.readthedocs.io/en/stable/api.html 、
  作者の回答 https://github.com/miguelgrinberg/Flask-SocketIO/discussions/2039 ）
- したがって保証されるのは**サーバー側のイベントロジックの正しさ**まで。
  実ネットワーク伝送（WebSocket）・ブラウザJS配線・実SQLite書き込みは保証されない。
  → 実機確認は**別ブラウザタブ2つ**で公開マッチを成立させる方法で行う必要がある。

## 3. AI/人の優先度ルール（紙上設計・値は未確定）

> **注**: この節の具体値や方式はまだ確定していない提案段階。核心の考え方は確定に値する。

### 適用範囲

「待ち時間・補填確率」が効くのは主にクイックマッチ（公開）。シングルとプライベートは
構成ルールが既決なので、詰めるべきはクイックのみ。

### 設計の核: 2つの目的を分離する

- **(A) 構成ダイヤル（コスト・データ量の調整）**
  - `α` … 部屋成立時に相手がAIである基準確率
  - `T` … 人間を待つ最大秒数
  - `q` … タイムアウト時にAIで補填する確率
  - 初期提案値: α=0.5 / T=20秒 / q=1.0
  - 副作用: q=1.0 は低トラフィック時に実効AI率がαより上がる。
    コスト（Gemini無料枠）を重視するなら α↓ / T↑ / q<1 で人優先に倒す。

- **(B) 整合性の制約（研究の妥当性を守る、譲れない条件）**
  - プレイヤーが観察できるのは「待ち時間」と「会話」だけ。
  - 決定論的補填（空き枠は即AI）にすると、「遅く繋がった＝AI」という手がかりで研究が汚染される。
  - 対策1: 相手がAIのときも、人間マッチと同じ分布からサンプルした遅延で開始し、
    開始時間の分布を相手の正体によらず重ねる。
  - 対策2（役割の漏れ）: 非対称型でAIが必ず回答者だと「自分が回答者＝相手は人間」が
    完全な手がかりになる。クイックでは常に「判定者＝質問者／被験対象＝回答者」と固定し、
    役割が正体情報を持たないようにする。

### 決めるべき4点（未確定）

1. α（AIの基準確率）を0.5でいくか、コスト優先で下げるか
2. T（人間を待つ秒数）の値
3. q（タイムアウト時の補填確率）を1.0にするか、1未満にして再マッチに回すか
4. 整合性2ルール（開始タイミングの独立性・役割固定）を採用するか

## 4. 未実装・課題（優先度順）

1. **AI/人の優先度ルールの確定**（上記4点をユーザーが決定 → 確定仕様に書き起こし）
2. **グループの本実装**
   - クイックマッチのAI補填ロジック（優先度ルール込み）
   - **プライベートの合言葉照合**: 現状サーバー・フロントとも未実装。
     合言葉は入力欄と「空欄でないか」チェックのみで、`join_private` 等のサーバー処理は存在しない。
     本実装には「部屋コード→合言葉・参加者を持つ部屋台帳」＋入室時照合が必要。
3. **オンラインのアンケート／注釈の配線**
   - 既存の `/survey` `/annotate` はセッションの `game_id` を使うため、オンラインの
     `game_id`（部屋側）と繋がらない。オンライン経路では自己申告データが現状取れていない。
4. **注釈UI（フロントエンド）**: サーバー側のみ実装済み、UIは未実装。
5. **大人数型（最大10人）の進行管理**
6. **データ量対策**: 重い会話ログはCSV/JSONへ書き出してアーカイブ、DBは集計用に軽く保つ。

### 既知の技術的制約

- サーバーを**複数ワーカーで動かすと部屋を共有できない**（状態がプロセス内メモリのため）。
  スケール時は Redis 等のメッセージキューが必要。
  （出典: Flask-SocketIO "Deployment" https://flask-socketio.readthedocs.io/en/stable/deployment.html ）
  被験者規模なら当面は単一プロセスで十分。
- `async_mode="threading"` は導入が簡単な代わりに本番性能は要検討。

### 研究完全性のリスク（再掲・最重要）

決定論的なAI補填により、プレイヤーが「接続タイミング」を判別ヒューリスティックとして使えてしまうと、
研究のコア設問（会話のどんな発言でAIを見破るか）が汚染される。上記 3-(B) の整合性制約で対処する。
